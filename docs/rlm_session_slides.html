Bridge

From code execution to Recursive Language Models

1. Ask Model gets a task it cannot solve reliably in one shot.
2. Compute CodeAgent lets it use Python for exact steps.
3. Split RLM lets it issue smaller prompts with llm_call (direct) or rlm_call (recursive).
4. Merge Parent collects local answers into one global answer.
Tool use solves exact substeps. RLM adds self-decomposition across multiple reasoning stages.
Why it matters

Why recursion helps

  • Some tasks are too long to fit comfortably in one context window
  • Some tasks are easier if we solve local subproblems first
  • Some tasks benefit from a tree of intermediate results instead of one long prompt

Mental model

Think of it as divide-and-conquer for inference: split, solve smaller pieces, then aggregate.

This is especially useful for search, summarisation, long-context reasoning, and structured analysis.

RLMs trade a single large reasoning jump for a sequence of smaller ones.
Core idea

Standard LLM call vs. RLM call

Mode What the model can do Main limitation
Plain LLM Answer directly from a single prompt Context window and one-shot reasoning limit
CodeAgent Write Python to compute or inspect intermediate results Still solving one logical task frame at a time
RLM Write code and recursively call child agents on subprompts Needs orchestration, depth limits, and traceability
RLM is not a different model. It is a different inference pattern.
Two tools

llm_call vs. rlm_call

Tool Behaviour Use when
llm_call(sub_task, context_slice) Direct, non-recursive LLM call — no child REPL spawned Chunk is already small enough to answer in one shot
rlm_call(sub_task, context_slice) Recursive sub-agent — child gets its own REPL and tools Sub-task may itself need further decomposition

The model decides freely in Python how to orchestrate — it can split by paragraphs, by regex, by length, or any other logic before choosing which tool to call.

Mirrors the paper's llm_query / rlm_query distinction.

llm_call = fast leaf call. rlm_call = recursive sub-agent. The model orchestrates freely.
Mechanism

How RLMAgent works internally

completion(prompt)
create root node
↓
_run(...)
build CodeAgent
llm_call(...)
direct leaf call
rlm_call(...)
spawn sub-agent
↓
child answer A
child answer B
fallback leaf
1. The root call creates the first node in the tree.
2. The agent decides whether to answer directly or delegate subproblems.
3. llm_call: direct answer for small chunks. rlm_call: new child agent with its own REPL.
4. The parent aggregates all child responses into the final answer.
The same pattern can be understood without the notebook: build agent, recurse, aggregate, trace.
Walkthrough

A concrete example flow

Root agent Splits the corporate report into 6 sections using Python.
Child 1–6 Each calls llm_call() to summarise or comprehend its section.
Root aggregation Collects section summaries/answers, combines into final output.
Output Returns a structured executive summary or answered questions.
This mirrors the RLM paper's core demonstration: recursive decomposition for tasks that exceed single-shot LLM capacity.

Showcase

Recursive summarization & comprehension Q&A

Notebook 03 demonstrates what makes RLM fundamentally different from a plain LLM call — the agent must decompose and delegate.

ExperimentWhy sub-agents are required
A — Hierarchical SummarizationSummarization is an LLM task — you cannot summarize with regex. The agent splits the report into sections and calls llm_call on each.
B — Natural-Language Q&AFacts are embedded in natural prose (numbers as words, no labels). Sub-agents must read and comprehend each section.

Document: ~6,000-word multi-section corporate report with 6 sections, ~90 paragraphs, 5 comprehension questions. Uses block-format SECTION: separators for reliable splitting.

Why this is hard for a plain LLM

A single-shot LLM must process the entire document at once. The RLM splits by section, delegates comprehension to sub-agents, and aggregates results — turning a reading-comprehension problem into a recursive search algorithm. Regex shortcuts are impossible because facts are written as natural prose.

Lessons learned

Document format & split validation

The problem

An early version of Notebook 03 used === Name === section separators. The agent wrote split("===") and got 12 fragments (alternating headers and content) instead of 6 sections — cascading into garbled sub-agent answers.

Two-part fix (Option A + D)

FixWhat changed
A — Unambiguous separators Sections now use SECTION: Name between lines of = characters. Splitting on SECTION: is straightforward and reliable.
D — Split-validation prompt The system prompt tells the agent: “After splitting, print the number of chunks and a short preview of each. If it looks wrong, fix it before making sub-calls.”

Takeaway

The agent already has all the tools it needs (Python REPL). The fix is better input design (clear separators) plus defensive prompting (validate before delegating). No extra parsing libraries required.

Good separator design + split validation = robust recursive decomposition.

Closing

One-sentence summary

A Recursive Language Model is an inference pattern where the model can write code, break a task into subproblems, call itself on those subproblems, and then combine the results into a final answer.

Plain LLM: one shot. CodeAgent: one shot plus tools. RLM: tools plus self-decomposition via llm_call (direct) and rlm_call (recursive).
