The motivation

Why not just keep increasing context windows?

Long prompts are expensive, brittle, and hard to steer
Single-shot long-context prompting
  • Everything is packed into one prompt string.
  • The model must find the right evidence while also reasoning about the task.
  • As prompts get longer, attention is diluted and irrelevant text competes with useful text.
  • Instructions and raw data live in the same channel.
RLM view
  • Use the model as an orchestrator, not just a one-shot responder.
  • Let it inspect data programmatically and focus only on relevant slices.
  • Turn one impossible call into a sequence of simpler, cheaper calls.
  • Separate instructions from data.

Intuition: if a human would skim, split, search, summarize, and combine, an RLM should be allowed to do the same.

Key concept

Context lives in the environment, not in the prompt

This is the central shift
Traditional call 
response = llm.completion(prompt)
# prompt contains task + document text

The raw document is copied into the prompt every time.

RLM call 
response = rlm.completion(task, context)
# context is stored externally

The model receives a task plus tools for inspecting rlm_context.

Data channel

rlm_context is a Python variable in the REPL.

Instruction channel

The prompt describes the task and available tools, not the full document.

Result

Sub-calls can operate on slices without repeatedly re-embedding the entire context.

How it works

An RLM solves tasks by decomposition

The model writes code to orchestrate the process
1

Receive task + context. The system injects the context into the execution environment as rlm_context.

2

Inspect and split. The model can slice, search, or divide the context using normal Python.

3

Launch child calls. Small factual lookups can use direct LLM calls; complex sub-problems can recurse again.

4

Aggregate answers. The parent combines intermediate results into a final response.

Minimal example

The pattern is inspect → split → query → merge

The full document never becomes a child prompt string 
mid = len(rlm_context) // 2
left = rlm_call("Summarize first half", rlm_context[:mid])
right = rlm_call("Summarize second half", rlm_context[mid:])
final_answer(left + " " + right)
What matters
  • The parent can choose a splitting strategy dynamically.
  • Each child sees only a smaller slice and a narrower task.
  • The decomposition can continue until the slice is simple enough.
  • Avoid: building child prompts with the entire raw context interpolated into strings.

Two kinds of sub-calls

Leaf queries and recursive queries play different roles

The repo mirrors the paper’s direct vs recursive distinction
Direct leaf call 
llm_call(sub_task, context_slice)
  • Fast, one-shot answer on a small slice.
  • Best for extraction, summarization, or local reasoning.
  • Used when the problem is already reduced enough.
Recursive call 
rlm_call(sub_task, context_slice)
  • Creates another RLM node with its own REPL state.
  • Useful when a child still needs search, splitting, or aggregation.
  • Builds a call tree instead of a flat prompt.

Why this is interesting

RLMs change both scale and robustness

Not just bigger context — better structure
Scalability
  • Operate over data larger than one prompt should reasonably contain.
  • Reuse the same orchestration pattern across summarization, QA, retrieval, and synthesis.
  • Keep each model call focused and local.
Robustness
  • Data is treated as data, not instruction text.
  • Separating context from the prompt reduces accidental steering by hostile text embedded in documents.
  • Explicit split validation helps prevent bad chunking from cascading into bad answers.

Takeaway: the win is not only “more tokens.” The win is giving the model a structured way to work over external state.
