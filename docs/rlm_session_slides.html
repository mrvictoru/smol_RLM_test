Warm-up demo

Why start with letter counting?

Ask the audience:

How many times does the letter "r" appear in "strawberry"?

This looks trivial, but token-based generation is not the same thing as exact character-level computation.

What this demonstrates

  • LLMs are strong at pattern completion, not guaranteed exact counting
  • Small symbolic tasks can expose brittle reasoning
  • This makes a strong setup for showing why external tools matter

Tool use

Let the model write Python instead of guessing

count_word = "strawberry" target_letter = "r" count_prompt = """ Count how many times the letter 'r' appears in 'strawberry'. Important: do not answer from intuition. Use Python code in the REPL to compute the answer exactly. """ count_result = count_agent.completion(count_prompt)

Message to the audience

  • The improvement is not magic. We changed the execution model.
  • The LLM can externalize the hard part into code.
  • The answer becomes deterministic because Python is doing the counting.

Bridge

From code execution to Recursive Language Models

Step 1 Plain LLMs try to solve everything in one pass inside a token stream.
Step 2 A CodeAgent lets the model write Python when exact computation helps.
Step 3 An RLM lets the model create smaller subproblems with rlm_call(sub_prompt).
Step 4 Parent and child calls form a recursive tree that can be inspected after the run.
Tool use solves exact substeps. RLM adds self-decomposition.
Core idea

Standard LLM call vs. RLM call

Mode What the model can do Main limitation
Plain LLM Answer directly from a single prompt Context window and one-shot reasoning limit
CodeAgent Write Python to compute or inspect intermediate results Still solving one logical task frame at a time
RLM Write code and recursively call child agents on subprompts Needs orchestration, depth limits, and traceability
RLM is not a different model. It is a different inference pattern.
Mechanism

How RLMAgent works internally

  1. completion(prompt) creates the root call node.
  2. _run(...) builds a CodeAgent and injects the rlm_call tool.
  3. The model decides whether to answer directly or delegate subproblems.
  4. Each child call creates its own node in the recursive tree.
  5. If the depth limit is reached, the code falls back to a plain completion.
  6. The tree and all traced prompts are returned in metadata.
completion(prompt) -> _run(prompt, depth=0) -> CodeAgent(REPL + rlm_call) -> rlm_call(sub_prompt) -> _run(sub_prompt, depth + 1) -> or _plain_completion(sub_prompt) -> aggregate child results -> return response + call_tree + llm_requests
Use the notebook trace cells to make this concrete.
Observability

Prompt tracing makes the agent explainable

With capture_prompt_traces=True, you can inspect the exact messages sent to the LLM server at each step.

  • Root agent prompt
  • Child prompts from rlm_call
  • Plain fallback payloads at max depth
trace_result = trace_agent.completion(trace_prompt) for request in trace_result.iter_llm_requests(): print(request["depth"], request["phase"]) for message in request["messages"]: print(message["role"], message.get("content", ""))
This is useful for demos, debugging, and explaining the mechanism live.

Why this order works

The audience sees a concrete reliability win first. Once they accept that the model can use external execution, recursive self-decomposition feels like a natural next step instead of a theoretical abstraction.

Closing

One-sentence summary

A Recursive Language Model is an inference pattern where the model can write code, break a task into subproblems, call itself on those subproblems, and then combine the results into a final answer.

Plain LLM: one shot. CodeAgent: one shot plus tools. RLM: tools plus self-decomposition.
